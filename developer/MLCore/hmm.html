Hidden Markov Model (Gaussian / GMM)

Gaussian HMM for temporal sequence classification via mlpack, with optional mixture-of-Gaussians emissions

Overview

HiddenMarkovModelOperation wraps mlpack::HMM<Distribution> to provide sequence-aware classification within the MLCore framework. Unlike frame-based classifiers (Random Forest, Logistic Regression) that treat each observation independently, the HMM exploits temporal continuity — the probability of being in state \(s_t\) depends on \(s_{t-1}\).

Four emission distribution variants are supported:

Variant mlpack type Use case
Full Gaussian (default) HMM<GaussianDistribution<>> Unimodal emissions with full covariance
Diagonal Gaussian HMM<DiagonalGaussianDistribution<>> Unimodal with diagonal covariance (fewer parameters)
GMM HMM<GMM> Multi-modal emissions with full covariance per component
Diagonal GMM HMM<DiagonalGMM> Multi-modal with diagonal covariance per component

Files:

  • Header: src/MLCore/models/supervised/HiddenMarkovModelOperation.hpp
  • Implementation: src/MLCore/models/supervised/HiddenMarkovModelOperation.cpp
  • Tests: tests/MLCore/HiddenMarkovModel.test.cpp

Architecture

Sequence API on MLModelOperation

The base MLModelOperation class provides optional virtual methods for sequence-aware models:

Method Default Behavior HMM Override
trainSequences(featureSeqs, labelSeqs, params) Concatenates sequences, calls train() Passes sequences directly to mlpack::HMM::Train()
predictSequences(featureSeqs, predSeqs) Concatenates, calls predict(), splits back Viterbi decodes each sequence independently
predictSequencesConstrained(featureSeqs, predSeqs, constraints) Ignores constraints, delegates to predictSequences() Clamps hmm.Initial() per-sequence, Viterbi decodes, restores
isSequenceModel() Returns false Returns true

Non-sequence models (Random Forest, etc.) inherit the default concatenation behavior, so existing code is unaffected.

Pimpl Pattern

The implementation follows the project-standard Pimpl pattern to keep mlpack.hpp out of the header. A std::variant holds one of the four HMM types, with all operations dispatched via a visitor:

struct HiddenMarkovModelOperation::Impl {
    std::variant<std::unique_ptr<GaussianHMM>,
                 std::unique_ptr<DiagonalGaussianHMM>,
                 std::unique_ptr<GMMEmissionHMM>,
                 std::unique_ptr<DiagonalGMMEmissionHMM>> model;
    std::size_t num_states = 0;
    std::size_t num_features = 0;
    std::size_t num_gaussians = 0;
    bool trained = false;
    bool diagonal = false;
    bool gmm_emissions = false;
};

Training

Sequence-Based (Preferred)

The trainSequences() method receives pre-segmented temporal sequences (typically from SequenceAssembler) and calls mlpack::HMM::Train() directly:

HiddenMarkovModelOperation hmm;
HMMParameters params;
params.num_states = 2;

// featureSeqs: vector of (features × seq_length) matrices
// labelSeqs: vector of (1 × seq_length) row vectors (state indices)
hmm.trainSequences(featureSeqs, labelSeqs, &params);

Validation checks:

  • Non-empty sequence vectors
  • Feature/label sequence count match
  • Column/label count match within each sequence
  • Consistent feature dimensionality across sequences

Frame-Based Fallback

The train() method wraps the input as a single sequence and delegates to trainSequences():

hmm.train(features, labels, &params);  // Equivalent to single-sequence training

Prediction

Viterbi Decoding

predict() and predictSequences() use mlpack’s Viterbi algorithm to find the most probable hidden state sequence:

arma::Row<std::size_t> predictions;
hmm.predict(features, predictions);

// Or per-sequence:
std::vector<arma::Row<std::size_t>> predSeqs;
hmm.predictSequences(featureSeqs, predSeqs);

Forward-Backward Probabilities

predictProbabilities() returns per-frame state probabilities via the Forward-Backward algorithm:

arma::mat probs;  // (num_states × observations)
hmm.predictProbabilities(features, probs);
// probs(state, time) = P(state at time t | all observations)

Constrained Viterbi Decoding

predictSequencesConstrained() overrides the HMM’s initial state distribution on a per-sequence basis before running Viterbi. This is used to warm-start prediction when a boundary frame’s state is known from training labels.

// constraints[i] = known initial state for sequence i, or nullopt
std::vector<std::optional<std::size_t>> constraints{std::size_t{0}, std::nullopt};
std::vector<arma::Row<std::size_t>> preds;
hmm.predictSequencesConstrained(featureSeqs, preds, constraints);

Implementation: saves hmm.Initial(), sets it to a delta distribution [0,...,1,...,0] on the constrained state, calls Predict(), and restores. This ensures subsequent unconstrained predictions are unaffected. The ClassificationPipeline builds a label map from training data and automatically derives constraints when constrained_decoding is enabled.

Each column sums to 1.0.

Introspection

Transition Matrix

arma::mat trans = hmm.getTransitionMatrix();
// trans(i, j) = P(state j at t+1 | state i at t)
// Columns sum to 1.0

Log-Likelihood

double ll = hmm.logLikelihood(features);
// Returns -infinity if not trained or dimensions mismatch

Parameters

HMMParameters (models/MLModelParameters.hpp):

Parameter Default Description
num_states 2 Number of hidden states (minimum 2)
tolerance 1e-5 Convergence tolerance for training
use_diagonal_covariance false Use diagonal covariance matrices for emissions
use_gmm_emissions false Use Gaussian Mixture Model emissions per state
num_gaussians 3 Number of Gaussian components per state (when use_gmm_emissions is true, minimum 2)

Emission Type Selection

The use_diagonal_covariance and use_gmm_emissions flags combine orthogonally:

use_gmm_emissions use_diagonal_covariance Emission type
false false Single Gaussian (full covariance)
false true Single Gaussian (diagonal covariance)
true false GMM (full covariance per component)
true true Diagonal GMM (diagonal covariance per component)

GMM Emissions

When use_gmm_emissions is true, each hidden state uses an mlpack::GMM (or mlpack::DiagonalGMM) as its emission distribution instead of a single Gaussian. This allows each state to model multi-modal observation distributions 014 useful when a single hidden state actually encompasses multiple sub-states with distinct emission characteristics.

Each state019s GMM has num_gaussians components. During supervised training, mlpack fits the mixture via EM on the observations assigned to that state. Prediction (Viterbi, Forward-Backward) works identically to the single-Gaussian case; only the emission probability computation changes.

The isGMMEmissions() and numGaussiansPerState() methods report the active configuration after training. Serialization includes the GMM flag and component count, so loaded models correctly reconstruct the matching HMM variant.

Diagonal Covariance

When use_diagonal_covariance is true, the HMM uses mlpack::DiagonalGaussianDistribution<> instead of full covariance. This:

  • Reduces covariance parameters per state from \(\frac{M(M+1)}{2}\) to \(M\)
  • Reduces emission evaluation cost from \(O(M^2)\) to \(O(M)\)
  • Improves numerical stability with limited training data
  • Assumes features are conditionally independent given the hidden state

The isDiagonalCovariance() method reports the active variant after training. Serialization includes the diagonal flag, so loaded models correctly reconstruct the matching HMM type.

Serialization

Save/load uses cereal binary archives, matching the pattern of other MLCore models:

std::stringstream ss;
hmm.save(ss);

HiddenMarkovModelOperation loaded;
loaded.load(ss);

Pipeline Integration

The ClassificationPipeline detects sequence models via isSequenceModel() and automatically activates the SegmentingSequences stage. The modified pipeline flow for HMM:

ValidatingFeatures → ConvertingFeatures → AssemblingLabels →
BalancingClasses → SegmentingSequences → Training → Predicting →
ComputingMetrics → WritingOutput → Complete

Sequence Mode Behavior

  1. After label assembly, SequenceAssembler::segment() splits the feature matrix and labels into contiguous temporal segments based on gaps in TimeFrameIndex.
  2. Training routes through trainSequences() instead of train().
  3. Prediction routes through predictSequences(), then reassembles flat predictions and time indices from the per-sequence results.
  4. Metrics and output writing operate on the reassembled flat vectors, identical to frame-based models.

Label Sources

The HMM works with all four label assembly modes. The most natural pairing is LabelFromEvents (point-event data → binary state labels), but LabelFromIntervals, LabelFromGroups, and LabelFromEntityGroups are equally supported.

// Event-based labels (ideal for HMM)
config.label_config = LabelFromEvents{"Contact", "NoContact"};
config.label_event_key = "contact_events";

// Interval-based labels (also works)
config.label_config = LabelFromIntervals{"Contact", "NoContact"};
config.label_interval_key = "contact_intervals";

End-to-End Example

ClassificationPipelineConfig config;
config.model_name = "Hidden Markov Model (Gaussian)";

HMMParameters hmm_params;
hmm_params.num_states = 2;
config.model_params = &hmm_params;

config.feature_tensor_key = "whisker_features";
config.label_config = LabelFromEvents{"Contact", "NoContact"};
config.label_event_key = "contact_events";
config.conversion_config.zscore_normalize = true;
config.prediction_region.predict_all_rows = true;
config.output_config.output_prefix = "HMM:";
config.time_key_str = "camera";

auto result = runClassificationPipeline(dm, registry, config, progress_cb);

if (result.success) {
    // result.trained_model is HiddenMarkovModelOperation
    auto * hmm = dynamic_cast<HiddenMarkovModelOperation *>(result.trained_model.get());
    arma::mat trans = hmm->getTransitionMatrix();
    // trans(0,0) ≈ P(stay in state 0), trans(0,1) ≈ P(transition 0→1)
}

Widget Integration

The MLCore_Widget provides HMM-specific UI elements across three panels:

Model Configuration (ModelConfigPanel)

When “Hidden Markov Model (Gaussian)” is selected in the model combo box, the stacked widget switches to the HMM parameter page. Controls:

Widget Parameter Range
QSpinBox Number of states 2–100
QDoubleSpinBox Tolerance 1e-10 – 1.0
QCheckBox Diagonal covariance on/off
QCheckBox GMM emissions on/off
QSpinBox Gaussians per state 2–50 (enabled only when GMM is checked)

Label Configuration (LabelConfigPanel)

A fourth radio button, “From Events (ideal for HMM)”, enables selection of a DigitalEventSeries from the DataManager. This maps point events to binary state labels (event present → state 1, absent → state 0). State is persisted as label_event_key in MLCoreWidgetStateData.

Results (ResultsPanel)

After training, the results panel detects if the trained model is an HMM via dynamic_cast and displays the learned transition matrix in a monospaced formatted text block. The matrix shows \(P(\text{next state} \mid \text{current state})\) with columns summing to 1.0.

Limitations

  • Supervised only: The current implementation requires labeled sequences. Unsupervised Baum-Welch training (EM) is not exposed, though mlpack supports it.
  • Gaussian / GMM emissions: Four emission variants are supported (full/diagonal Gaussian, full/diagonal GMM). Discrete emissions are not supported.
  • Fixed state count: The number of states must be specified before training. Model selection (choosing the optimal state count) must be done manually.
  • No warm-starting: Each train() / trainSequences() call initializes a fresh HMM. Incremental refinement of an existing model is not supported.
  • Memory: The Forward-Backward algorithm allocates a (num_states × observations) probability matrix. Very long sequences with many states may consume significant memory.
