Scatter Plot Widget Roadmap

ScatterPlotWidget Development Roadmap

This document tracks remaining development tasks for the Scatter Plot Widget. For documentation of completed features and current architecture, see the main developer documentation.

Not yet implemented: Stale key handling (Phase 6.3).

Phase 7: External Feature Coloring (Widget Integration)

Status: Complete. All widget integration steps finished.

Goal: Color scatter points by an external data source (e.g., velocity colored by acceleration, points colored by whether they fall inside a DigitalIntervalSeries interval).

Completed Infrastructure

The shared CorePlotting/FeatureColor/ library is fully implemented and tested (24 test cases, 74 assertions). See the FeatureColor module docs for the full API reference. Key components:

  • FeatureColorSourceDescriptor — serializable pointer to a DataManager key + optional column
  • resolveFeatureColors() — per-point float lookup via ATS, TensorData (TFI hash join / ordinal positional), or DIS containment
  • checkFeatureColorCompatibility() — validates key existence, column existence, row-type compatibility
  • applyFeatureColorsToScene() — mutates RenderableGlyphBatch::colors[] via ContinuousMapping (colormap) or ThresholdMapping (binary)
  • computeEffectiveColorRange() — Auto/Manual/Symmetric vmin/vmax from resolved values
  • PlotSourceRowType — shared enum (replaces the earlier ScatterSourceRowType)

Rendering Pipeline

Feature coloring is a post-buildScatterPoints() color override inserted before group colors:

buildScatterPoints()          → ScatterPointData with time_indices
    → addGlyphs()             → uniform color from GlyphStyleData
    → applyFeatureColorsToScene()  → per-point color override (NEW)
    → applyGroupColorsToScene()    → group colors override feature colors
    → uploadScene()

No changes needed to SceneBuilder, GlyphRenderer, or shaders. Color priority (highest wins): navigate highlight > selected > group > feature > default glyph style.

7.3 Rendering Integration

In ScatterPlotOpenGLWidget::rebuildScene(), after builder.build() and before applyGroupColorsToScene():

auto const & cc = _state->colorConfig();
if (cc.color_mode == "by_feature" && cc.color_source.has_value()) {
    auto const feature_values = resolveFeatureColors(
        *_data_manager, *cc.color_source,
        _scatter_data.time_indices, _scatter_data.time_frame);

    FeatureColorMapping mapping;
    if (cc.mapping_mode == "threshold") {
        mapping = ThresholdMapping{
            static_cast<float>(cc.threshold),
            hexColorToVec4(cc.above_color, cc.above_alpha),
            hexColorToVec4(cc.below_color, cc.below_alpha)};
    } else {
        auto const range = computeEffectiveColorRange(feature_values, cc.color_range);
        if (range.has_value()) {
            auto const preset = colormapPresetFromString(cc.colormap_preset);
            mapping = ContinuousMapping{preset, range->first, range->second};
        }
    }

    applyFeatureColorsToScene(*_scene, feature_values, mapping, point_count);
}

applyGroupColorsToScene();  // Group colors override feature colors

Cache feature_values as a member for tooltip access (step 7.4).

Design Decisions

  1. Post-build color override: Feature colors mutate RenderableGlyphBatch::colors[] after addGlyphs(), matching the existing applyGroupColorsToScene() pattern. No SceneBuilder or GlyphRenderer changes needed.

  2. Threshold mode as generalized binary coloring: A single threshold mapping handles DIS (0/1 at threshold=0.5), binary classifier output, and continuous cutoffs (e.g., “velocity > 20”). It is a peer of continuous colormap mode, not a special case.

  3. Shared CorePlotting/FeatureColor/ library: Resolution and rendering logic is shared with EventPlotWidget (Phase 5) and LinePlotWidget (Phase 3). Each widget provides its own time_indices and integrates the color config into its EditorState.

Architecture Assessment

The ScatterPlotWidget rendering pipeline is the most compatible of all remaining widgets:

  • It already caches a RenderableScene as _scene (std::unique_ptr<CorePlotting::RenderableScene>)
  • The scene contains RenderableGlyphBatch entries (circles, squares, crosses for scatter points) and optionally a RenderablePolyLineBatch (reference line). Both are fully supported by SVGGlyphRenderer and SVGPolyLineRenderer.
  • Feature colors and group colors are pre-applied to RenderableGlyphBatch::colors[] during rebuildScene(), so the SVG renderer sees the final per-point colors with no additional work.

Background color: ScatterPlotStateData includes background_color (default #1A1A1A, serializable). ScatterPlotOpenGLWidget uses updateBackgroundColor() (CoreUtilities::hexToRGB) in initializeGL() and paintGL(), and listens for backgroundColorChanged(). A properties-panel color control (like EventPlot) is optional and not required for SVG export.

Step 8.2: Add exportToSVG() to ScatterPlotOpenGLWidget — Complete

Follows the EventPlotOpenGLWidget pattern. The scatter plot caches _scene as std::unique_ptr<RenderableScene>; live view/projection matrices are copied onto the scene before export (the OpenGL path passes matrices into SceneRenderer::render(), so the CPU scene would otherwise keep identity defaults).

Declaration: [[nodiscard]] QString exportToSVG(); (non-const — mutates scene matrices for export).

Implementation (reference):

QString ScatterPlotOpenGLWidget::exportToSVG() {
    if (!_scene) {
        return {};
    }
    if (_scene->glyph_batches.empty() && _scene->poly_line_batches.empty() &&
        _scene->rectangle_batches.empty()) {
        return {};
    }

    _scene->view_matrix = _view_matrix;
    _scene->projection_matrix = _projection_matrix;

    PlottingSVG::SVGSceneRenderer renderer;
    renderer.setScene(*_scene);
    renderer.setCanvasSize(_widget_width, _widget_height);

    if (_state) {
        renderer.setBackgroundColor(_state->getBackgroundColor().toStdString());
    }

    return QString::fromStdString(renderer.render());
}

Step 8.6: Manual validation — Complete

Checklist used for sign-off (spot-check as needed for regressions):

  1. Load a dataset with scatter-compatible data (e.g., TensorData or paired AnalogTimeSeries)
  2. Open a ScatterPlotWidget, configure X/Y data sources
  3. Test with:
    • Default uniform glyph coloring (circles)
    • Feature coloring (continuous colormap mode)
    • Feature coloring (threshold mode)
    • Group coloring (color-by-group)
    • Reference line enabled
    • Point selection (polygon selection or single-click)
  4. Click “Export SVG” in the properties panel
  5. Open the resulting SVG in a web browser and/or Inkscape
  6. Verify:
    • Glyph positions match on-screen points
    • Per-point colors (feature/group) are correct
    • Selected points have highlight color
    • Reference line rendered as <polyline> (if enabled)
    • Background color is correct
    • Zoom/pan state is reflected

Design Notes

  • No PlottingSVG extensions needed: RenderableGlyphBatch (circles, squares, crosses, ticks) and RenderablePolyLineBatch (reference line) are both fully supported.
  • Selection highlights included: Selected and navigated-highlight glyphs are separate GlyphBatch entries in the scene with their own colors/sizes, so they export automatically.
  • Polygon selection overlay NOT exported: The PolygonInteractionController preview is rendered via PreviewRenderer (not part of the RenderableScene), so the in-progress polygon outline will not appear in the SVG. This is desirable — the polygon is transient interaction state, not a data visualization element.
